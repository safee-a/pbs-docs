## Key Identifiers

| Field | Description | Found In |
|-------|-------------|----------|
| `SCHEDULE_CODE` | Monthly schedule identifier (e.g., `2026-02-01`) | All endpoints |
| `PBS_CODE` | Unique PBS item identifier | `items`, `fees`, `dispensing-rules`, `restrictions`, `item-overview`, `item-amt` |
| `ITEM_CODE` | Alternative item identifier | `items` |
| `PROGRAM_CODE` | PBS program identifier (e.g., `GE`, `RP`) | `items`, `programs`, `program-dispensing-rules` |
| `ATC_CODE` | WHO ATC classification code | `items`, `atc` |
| `RES_CODE` | Restriction identifier | `restrictions`, `parameters` |
| `PARAMETER_ID` | Restriction parameter identifier | `parameters`, `criteria` |
| `CRITERIA_ID` | Restriction criteria identifier | `criteria` |
| `PRESCRIBER_CODE` | Prescriber type code | `prescribers`, `restrictions` |

## Join Patterns

### Direct Foreign Key

Most relationships are direct foreign key joins. The field name matches exactly between endpoints:

```
items.PBS_CODE = dispensing-rules.PBS_CODE
items.PROGRAM_CODE = programs.PROGRAM_CODE
restrictions.RES_CODE = parameters.RES_CODE
```

### Hierarchical Navigation

The restriction system follows a strict hierarchy. Always traverse top-down:

```
restrictions (filter by PBS_CODE)
  → parameters (filter by RES_CODE)
    → criteria (filter by PARAMETER_ID)
```

### Composite Keys

Some relationships require multiple fields for uniqueness:

```
items: SCHEDULE_CODE + PBS_CODE (unique item in a schedule)
restrictions: SCHEDULE_CODE + RES_CODE (unique restriction in a schedule)
```
